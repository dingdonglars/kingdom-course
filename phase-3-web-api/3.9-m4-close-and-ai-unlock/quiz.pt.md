## 1. O AI Unlock passa a valer no...

- **a.** Fim da Fase 0 (a Spark Week + os Fundamentos)
- **b.** Fim da Fase 3 (API Web) — depois do M4. A partir daí, a IA pode ajudar com a implementação.
- **c.** No comecinho do currículo, antes de qualquer módulo
- **d.** A qualquer momento que o aluno decidir virar a chave

## 2. A regra rígida pós-unlock é...

- **a.** A IA deve sempre escrever o código para o aluno conseguir ler
- **b.** O aluno precisa conseguir explicar cada linha de código gerado pela IA antes de fazer o merge — a explicação é a regra do merge
- **c.** A IA não pode ser usada para nenhuma implementação em nenhuma circunstância
- **d.** Não há regra específica; é uma diretriz flexível

## 3. A chave de modo é virada editando...

- **a.** Uma linha numa tabela de banco de dados em algum lugar
- **b.** Uma única linha (`Current mode: pre-unlock` → `post-unlock`) no `CLAUDE.md` — no workshop, no curso e no starter-template
- **c.** Uma constante C# no `Program.cs`, e recompilando
- **d.** Uma interface de configuração na IDE

## 4. O modelo de PR pós-unlock adiciona uma seção de *assistência de IA*. O que vai nela?

- **a.** Só o nome do assistente de IA usado, nada mais
- **b.** O bot usado + quais linhas você escreveu versus quais a IA escreveu + qualquer coisa de que você não tem certeza
- **c.** Uma lista de bugs que a IA introduziu
- **d.** Comentários opcionais que nunca são lidos de verdade

## 5. Por que a lição chama a IA de *uma ferramenta elétrica, não um piloto automático*?

- **a.** Ferramentas elétricas exigem respeito. A disciplina que você construiu (engine versus camada externa, testes determinísticos, explicação como regra de merge) é o que te deixa usar a IA sem se perder.
- **b.** São metáforas exatamente equivalentes
- **c.** A IA é perigosa em todos os casos e deve ser evitada
- **d.** É uma tradição sem sentido real por trás
