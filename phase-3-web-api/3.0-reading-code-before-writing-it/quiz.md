## 1. Why read other people's code before writing your own?

- **a.** So you can copy it into your project
- **b.** Reading builds judgement — knowing why one structure is better than another. Tutorials teach syntax; reading teaches taste.
- **c.** Because the curriculum requires it
- **d.** To pass the time before the real work starts

## 2. What's roughly the right ratio of reading to writing for an experienced developer?

- **a.** About five times more reading than writing — they scan, find the right file, notice missing pieces
- **b.** Roughly equal — one to one
- **c.** Mostly writing; reading is for beginners
- **d.** Mostly reading; writing is for beginners

## 3. The five-question protocol asks you to find *one thing that surprises you*. Why?

- **a.** To make the journal entry longer
- **b.** The surprise is the gap between what you assumed and what's actually there — exactly the spot where new understanding lands
- **c.** It's a habit borrowed from book reviews
- **d.** Because xUnit requires it

## 4. Why does the lesson say *don't search for answers as you read*?

- **a.** To save bandwidth on slow connections
- **b.** Trying first, then checking, builds judgement. Searching immediately imports the answer without the cognitive stretch.
- **c.** Because the answers online are wrong
- **d.** To avoid breaking copyright

## 5. Reading, taking notes, and *committing* the notes — why is the commit part important?

- **a.** Reading-and-noting is real work; making it visible in your git log treats it like any other task and leaves a trail
- **b.** GitHub will reject commits without a journal entry
- **c.** It's a decoration — feel free to skip it
- **d.** To prepare for the M4 challenge
