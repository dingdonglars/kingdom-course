# Quiz answers — Module 3.0

## 1. b
Tutorials show you syntax — *write `await`, do `using`, use `Where`*. They don't teach why one shape of code is better than another. That comes from reading lots of real code and asking *"what does this person know that I don't?"* Reading builds judgement.

## 2. a
Seniors read *much* more than they write. They scan files quickly, know which folder to open first, recognise patterns. The 5x ratio is rough but real — most engineering work is reading and modifying, not creating from scratch.

## 3. b
The surprise is your edge of understanding. Where you expect X and see Y is exactly the place new knowledge fits. Writing it down forces you to articulate the gap; coming back later, you can verify whether the surprise resolved.

## 4. b
Trying first is the productive struggle that makes the answer stick. Googling immediately gives you the answer without the cognitive work — short-term win, long-term flat line. Try, then check; that's how judgement compounds.

## 5. a
Treat reading like any other deliverable. The commit is a public statement of what you did. The next session can refer back to your notes ("compare what we learned to what surprised you"). Plus: future-you in 3 months reading the git log will see "yep, I read that on day X."
