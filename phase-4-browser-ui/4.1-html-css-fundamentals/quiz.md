## 1. What's the difference between HTML and CSS?

- **a.** They do the same thing; the names are interchangeable
- **b.** HTML describes structure (sections, headings, lists); CSS describes appearance (colours, spacing, fonts)
- **c.** HTML handles input; CSS handles output for the page
- **d.** HTML runs on the server; CSS runs on the browser only

## 2. Why use semantic elements (`<header>`, `<main>`, `<nav>`) instead of `<div>` everywhere?

- **a.** They take fewer characters to type than a `<div>` with a class
- **b.** Meaningful tags help screen readers, search engines, and code readability
- **c.** They're a hard requirement of the HTML5 specification
- **d.** Browsers render them with built-in styles you can't change

## 3. What does `box-sizing: border-box` do?

- **a.** Adds a thin border around every element on the page
- **b.** Makes element width include padding and border, so `width: 200px` actually takes 200 pixels
- **c.** Removes the box layout entirely and uses inline rendering
- **d.** Required by every CSS file before other rules will apply

## 4. Why prefer `font-family: system-ui`?

- **a.** Native to each OS, no download, looks right by default everywhere
- **b.** It's the only font value the CSS parser will accept
- **c.** Older convention; modern projects use it for compatibility reasons
- **d.** Only Linux supports custom fonts, so this is the cross-OS fallback

## 5. The lesson says "markup first, style second, behaviour third." Why?

- **a.** Build semantic HTML that works with no CSS or JS, then style it, then add interactivity — skipping steps creates a mess
- **b.** It's a hard rule from the W3C web standards body
- **c.** It's measurably faster to write code in this order
- **d.** Search engines refuse to index sites that don't follow the order
