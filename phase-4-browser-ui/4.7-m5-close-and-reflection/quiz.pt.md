# Quiz — Módulo 4.7

> Tente primeiro o `quiz.md` em inglês. Use este aqui só quando uma palavra te travar.

## 1. Por que reler o seu código da Fase 0 no M5?

- **a.** Para ver o quanto você avançou — prova concreta de crescimento, não um sentimento
- **b.** Para rir do seu eu do passado e se sentir melhor com o código de hoje
- **c.** Exigido pelo currículo como parte do processo de revisão do PR de marco
- **d.** Para achar bugs que estavam latentes nos brinquedos originais da Fase 0 que você escreveu

## 2. Qual é a diferença entre *insatisfação genérica* e *insatisfação específica* com código antigo?

- **a.** Insatisfação genérica é ansiedade; insatisfação específica (*"eu extrairia este método"*) é domínio
- **b.** São a mesma coisa com dois nomes diferentes; a lição usa os dois de forma intercambiável
- **c.** A insatisfação genérica é mais saudável porque ainda não exige conhecimento técnico profundo
- **d.** A insatisfação específica é cruel com o seu eu do passado e deve ser evitada em exercícios de reflexão

## 3. Por que refatorar *um* brinquedo, e não a Fase 0 inteira?

- **a.** O objetivo é *ver o vão*, não apagar ele; uma refatoração prova que você consegue ver, refatorar tudo é trabalho de encheção
- **b.** O tempo é limitado e o currículo permite só trinta minutos para o exercício
- **c.** Exigido pelo Git — um commit de refatoração único por marco é a convenção aqui
- **d.** Os outros brinquedos não são ruins o suficiente para precisar de refatoração nesta etapa

## 4. Qual é a regra pós-Unlock que a descrição do PR tem que honrar?

- **a.** Seção de assistência de IA listando os arquivos tocados por IA; o autor consegue explicar cada linha
- **b.** Um link obrigatório para a transcrição da conversa de IA que produziu o código
- **c.** Um limite de tamanho de diff acima do qual PRs assistidos por IA não podem ser abertos
- **d.** Uma exigência de dois revisores em todo PR que inclui arquivos tocados por IA

## 5. Qual é o comando de tag do marco M5?

- **a.** `git push tag M5` e depois fazer merge com o branch master quando o revisor aprovar
- **b.** `git tag m5-phase-4-complete && git push origin m5-phase-4-complete`
- **c.** `git tag --milestone 5 && git milestone-push` a partir do branch de marco
- **d.** Não existe um; marcos agora são acompanhados só na entrada do log de vitórias
