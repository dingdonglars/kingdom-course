# Block 3 — Console Kingdom

> **Your own tycoon game runs in the terminal.**

**What you'll have at the end:** A playable console version of the Kingdom — buildings, resources, a ticking game loop, citizens, random events. You name your kingdom, play a 50-turn run, win or lose. With a real engine code library and your first unit tests.

**Estimated effort:** 9 weeks at 4–6 hrs/week  ·  **Wraps:** M2 *Kingdom v1 — Console*

---

## Why this block?

This is where the *Kingdom* — the mainstay project — actually starts. Every block from here forward is the same Kingdom in a different runtime: console, then persisted, then on the web, then in a browser, then on Roblox. The engine you build in this block is the engine you'll still be using in nine months.

This block also introduces two of the most important habits in real software: **engine vs shell** (the domain logic lives separately from the user interface) and **testing** (you write code that proves your code works). These come in the order they earn their place — not as theory.

## What you'll learn (named)

- Object-oriented programming: classes, properties, constructors, encapsulation
- The engine vs shell discipline (your first deliberate refactor)
- Unit testing with xUnit + Shouldly
- Inheritance and interfaces
- LINQ for querying your engine state
- Polymorphism via random events
- FakeItEasy for testing code that uses randomness or time
- Code organisation across multiple files and projects

## What you'll build (the climb)

- **Weeks 7–8:** OOP basics, then split the engine out from the console shell.
- **Week 9:** unit tests arrive; the game loop starts ticking.
- **Weeks 10–13:** building types, queries, random events, deterministic test fakes.
- **Weeks 14–15:** code organisation, then polish to a playable end-to-end Kingdom v1.

## Brag-worthy outcome

A playable game *you wrote*, that you can demo to anyone in 30 seconds. *"Yeah, I made a tycoon. Build farms, mines, deal with bandits, win or lose."*
