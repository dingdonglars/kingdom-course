## 1. The four sections of a good README are...

- **a.** Author / License / Changelog / Contact
- **b.** What / How to run / What you learned / What's next
- **c.** Install / Build / Test / Deploy
- **d.** Whatever the GitHub template puts there by default

## 2. What does `git stash` do?

- **a.** Permanently deletes uncommitted changes from your working tree
- **b.** Sets uncommitted changes aside on a stack — recover with `git stash pop`
- **c.** Pushes your changes to the remote without committing first
- **d.** Creates a new branch and switches to it in one step

## 3. The "rescue rule" is...

- **a.** Always force-push and start over from the remote
- **b.** Read the state (`git status`, `git log --oneline -10`) before acting on it
- **c.** Always `git reset --hard` to a known good commit
- **d.** Use the GUI; the command line is too risky for rescue work

## 4. Why is `git reset --hard` called *destructive*?

- **a.** It throws an error and refuses to run unless forced
- **b.** It permanently discards uncommitted changes (and possibly committed work, depending on what you reset to)
- **c.** It deletes the entire repository folder from disk
- **d.** It isn't destructive; the name is misleading

## 5. Why post a screenshot to `#wins`?

- **a.** Vanity — public credit for shipped work
- **b.** The before/after one-liner is what your future self reads in a year. The Slack post is the public commitment that makes it real.
- **c.** The curriculum requires it as part of grading
- **d.** It signals you're available for code review on the work
