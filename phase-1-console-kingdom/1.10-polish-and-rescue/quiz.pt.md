## 1. As quatro seções de um bom README são...

- **a.** Autor / Licença / Changelog / Contato
- **b.** O quê / Como rodar / O que você aprendeu / O que vem depois
- **c.** Instalar / Compilar / Testar / Implantar
- **d.** O que o template do GitHub colocar lá por padrão

## 2. O que `git stash` faz?

- **a.** Apaga permanentemente as mudanças não commitadas da sua árvore de trabalho
- **b.** Põe as mudanças não commitadas de lado, numa pilha — recupere com `git stash pop`
- **c.** Envia suas mudanças para o remoto sem commitar primeiro
- **d.** Cria um novo branch e troca para ele num passo só

## 3. A "regra do resgate" é...

- **a.** Sempre fazer force-push e recomeçar a partir do remoto
- **b.** Ler o estado (`git status`, `git log --oneline -10`) antes de agir sobre ele
- **c.** Sempre dar `git reset --hard` para um commit reconhecidamente bom
- **d.** Usar a interface gráfica; a linha de comando é arriscada demais para trabalho de resgate

## 4. Por que `git reset --hard` é chamado de *destrutivo*?

- **a.** Ele dá um erro e se recusa a rodar a menos que seja forçado
- **b.** Ele descarta permanentemente as mudanças não commitadas (e possivelmente trabalho já commitado, dependendo de para onde você resetar)
- **c.** Ele apaga a pasta inteira do repositório do disco
- **d.** Ele não é destrutivo; o nome engana

## 5. Por que postar uma captura de tela no `#wins`?

- **a.** Vaidade — crédito público pelo trabalho entregue
- **b.** A frase de antes/depois é o que o seu eu do futuro vai ler daqui a um ano. O post no Slack é o compromisso público que torna isso real.
- **c.** O currículo exige isso como parte da avaliação
- **d.** Sinaliza que você está disponível para revisão de código no trabalho
