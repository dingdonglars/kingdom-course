## 1. The five-thing read for AI code output is...

- **a.** Invented APIs, unhappy path, style drift, over-help, swallowed errors
- **b.** Bugs, performance, style, tests, docs
- **c.** Compile, run, test, ship, deploy
- **d.** Read, understand, accept, merge, forget

## 2. What's an invented API?

- **a.** A new feature added to the .NET runtime in a recent version
- **b.** A method call the AI generates confidently — but the method doesn't exist in your code or libraries
- **c.** A typo in a method name that the compiler will catch on the next build
- **d.** A C# version mismatch between the AI's training data and your project

## 3. Why is `catch { return null; }` a flag in AI-generated code?

- **a.** It swallows real errors silently — a network failure becomes "no kingdoms" rather than an exception you can act on
- **b.** It runs slower than catching a specific exception type
- **c.** Required to be removed by the .NET runtime
- **d.** Style preference; nothing actually wrong with it

## 4. Why is over-help a yellow flag in an AI response?

- **a.** The AI added behaviour you didn't ask for — either it's wrong (rolling back wastes time) or it's an unauthorised decision (scope creep)
- **b.** Over-help slows the AI down and increases token cost
- **c.** Required by the AI's training to add extra features by default
- **d.** Helpful AIs are bad in general; minimal output is always better

## 5. The lesson's standard isn't "perfect" — it's "output I can defend in the viva." What does that mean?

- **a.** You'll be asked to explain every line. Code you can explain ships; code you can't doesn't. The viva is the practical eval test.
- **b.** AI is too risky to use in real projects; only viva-graded code counts
- **c.** Required by the curriculum but not by the wider industry
- **d.** Style preference; "defend" is just a word the lesson likes
