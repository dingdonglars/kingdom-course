## 1. Qual é o recurso matador do painel Source Control do VS Code para commits caprichados?

- **a.** Ele gera as mensagens de commit automaticamente a partir das suas mudanças
- **b.** Ele faz o stage de arquivos inteiros mais rápido que o terminal
- **c.** Ele deixa você fazer o stage de linhas individuais (hunks) dentro de um arquivo
- **d.** Ele faz commit e push num clique só

## 2. O que o GitLens adiciona que o painel Source Control embutido não tem?

- **a.** Operações de stage e commit
- **b.** Blame em linha e uma visão de grafo visual dos commits
- **c.** Suporte a vários repositórios
- **d.** Botões de push e pull

## 3. Quando a linha de comando é mais rápida que o painel do VS Code?

- **a.** Para fazer stage e commit de mudanças rotineiras
- **b.** Para scripts, comandos escondidos como `reflog` ou `cherry-pick`, e por SSH
- **c.** Para ler um diff lado a lado
- **d.** Para ver o grafo de commits

## 4. Você precisa de um cliente gráfico dedicado, como o Fork ou o GitKraken?

- **a.** Sim — são as ferramentas profissionais padrão
- **b.** Só no Mac; usuários de Windows geralmente usam a linha de comando
- **c.** Não — a maioria dos desenvolvedores que trabalham na área usa a linha de comando mais o painel de git do editor
- **d.** Sim, se você já passou do AI Unlock; a interface gráfica é obrigatória pós-unlock

## 5. A regra de encerramento do B3 sobre linha de comando versus interface gráfica é:

- **a.** Escolha uma (linha de comando ou interface gráfica) e fique com ela; misturar é confuso
- **b.** Incline-se para a que for mais rápida para o movimento em questão; linha de comando e interface gráfica são um par
- **c.** Sempre use a linha de comando para produção; interfaces gráficas são para iniciantes
- **d.** Use a interface gráfica para branches compartilhados e a linha de comando para branches pessoais
