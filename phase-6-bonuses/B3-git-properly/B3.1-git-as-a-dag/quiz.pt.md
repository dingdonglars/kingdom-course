## 1. O que um commit do git de fato guarda?

- **a.** Um diff descrevendo o que mudou desde o último commit
- **b.** Um retrato do projeto inteiro num momento, mais uma mensagem e um ponteiro para o pai
- **c.** Só os arquivos que foram modificados, empacotados juntos
- **d.** Um arquivo de patch que pode ser aplicado a qualquer commit

## 2. O que é um branch do git?

- **a.** Uma cópia de todos os commits feitos naquele branch
- **b.** Uma pasta separada onde os commits daquele branch vivem
- **c.** Um ponteiro móvel para um commit específico
- **d.** Uma linha do tempo paralela de vida longa que é mesclada de volta periodicamente

## 3. Para onde o HEAD aponta, normalmente?

- **a.** O commit mais recente do `main`, não importa onde você esteja
- **b.** O branch em que você está agora (que, por sua vez, aponta para um commit)
- **c.** O branch de rastreamento remoto do branch atual
- **d.** Uma lista de todo commit recente que você fez

## 4. O que é o estado de *HEAD desanexado* (detached HEAD)?

- **a.** Um bug no git que precisa ser corrigido com `git reset`
- **b.** Quando o branch remoto e o local divergiram
- **c.** Quando o HEAD aponta direto para um commit em vez de para um branch
- **d.** Quando você tem mudanças não commitadas bloqueando um checkout

## 5. Por que o git usa hashes SHA como IDs de commit?

- **a.** São mais curtos do que números crescentes seriam
- **b.** São calculados a partir do conteúdo do commit, então qualquer mudança produz um hash novo
- **c.** São IDs aleatórios atribuídos pelo GitHub quando o commit recebe push
- **d.** São usados para criptografar o conteúdo do commit
