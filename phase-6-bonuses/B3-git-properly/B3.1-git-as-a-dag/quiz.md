## 1. What does a git commit actually store?

- **a.** A diff describing what changed since the last commit
- **b.** A snapshot of the whole project at one moment, plus a message and a parent pointer
- **c.** Only the files that were modified, packed together
- **d.** A patch file that can be applied to any commit

## 2. What is a git branch?

- **a.** A copy of all the commits made on that branch
- **b.** A separate folder where commits on that branch live
- **c.** A movable pointer to one specific commit
- **d.** A long-lived parallel timeline that gets merged back periodically

## 3. What does HEAD point at, normally?

- **a.** The most recent commit on `main`, regardless of where you are
- **b.** The branch you're currently on (which itself points at a commit)
- **c.** The remote tracking branch for the current branch
- **d.** A list of every recent commit you've made

## 4. What is *detached HEAD* state?

- **a.** A bug in git that needs to be fixed with `git reset`
- **b.** When the remote and local branches have diverged
- **c.** When HEAD points directly at a commit instead of at a branch
- **d.** When you have uncommitted changes blocking a checkout

## 5. Why does git use SHA hashes for commit IDs?

- **a.** They're shorter than incrementing numbers would be
- **b.** They're computed from the commit's contents, so any change produces a new hash
- **c.** They're random IDs assigned by GitHub when the commit is pushed
- **d.** They're used to encrypt the commit content
